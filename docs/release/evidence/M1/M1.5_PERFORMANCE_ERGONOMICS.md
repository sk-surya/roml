**Date:** 2026-07-14

## Benchmark Categories

| Category | What to measure | Tool |
|----------|----------------|------|
| Model build | Variable/constraint/cell creation throughput | Criterion |
| Parameter propagation | Parameter update → coefficient cascade timing | Criterion |
| Delta compilation | Model mutation → DeltaBatch generation | Criterion |
| Native apply | DeltaBatch → HiGHS state (per-operation timing) | Criterion |
| Snapshot rebuild | Full state rebuild from ModelSnapshot | Criterion |
| Solve | LP/MIP solve time (small/medium/large) | Criterion |
| Extraction | Solution value/dual/reduced-cost extraction | Criterion |

## Fixed Datasets

- `small`: 100 vars, 100 constraints, 1000 cells (dense-ish LP)
- `medium`: 1000 vars, 1000 constraints, 10000 cells (sparse LP)
- `large`: 10000 vars, 10000 constraints, 100000 cells (sparse LP)
- `mip_small`: Small MIP with binary/integer variables
- `parameterized`: Model with 50 parameters, 200 cells each (parameter propagation stress)

All benchmarks use fixed random seeds and report machine metadata (CPU, RAM, OS).

## Bulk Projection APIs

Where profiling shows FFI-call dominance (many small Highs_changeCoeff calls),
introduce bulk APIs:
- `set_cells_bulk(cells: &[(CellKey, f64)])` — single FFI batch for multiple coefficient changes
- `add_variables_bulk(vars: &[(Bounds, VarType)])` — batch variable addition
- `add_constraints_bulk(cons: &[(ConstraintBounds, Vec<(VarId, f64)>)])` — batch constraint addition

Bulk APIs must pass the same commuting-square tests as scalar APIs.
