## 1. Binding Decision Recap

The Xpress binding decision document ([`XPRESS_BINDING_DECISION.md`](../../XPRESS_BINDING_DECISION.md)) identifies three candidate approaches for integrating the FICO Xpress C API into Rust:

### Option A: Generated `roml-xpress-sys` with `links = "xprs"`

Generate Rust FFI declarations from official Xpress C headers using `bindgen` at build time. The sys crate owns the ABI contract and `links` key.

- **Pros:** ABI derived from installed headers matching the user's SDK version; no committed derivative code from proprietary headers; `links = "xprs"` ownership is clear; standard Cargo native-library pattern.
- **Cons:** Requires `bindgen` (LLVM dependency) at build time; Xpress headers must be installed and discoverable; licensing: header-derived bindings may still constitute derivative work; build complexity increases for users without Xpress.

### Option B: Runtime-loaded adapter with `libloading`

Load `libxprs` dynamically at runtime using `libloading`. No link-time dependency, no build-time header processing.

- **Pros:** No build dependency on Xpress SDK; clean compilation on hosts without Xpress installed; clear diagnostics when library is absent at runtime.
- **Cons:** Function pointers must be loaded and checked at runtime; struct layouts must still be defined (copied or generated); callback ABI from C into Rust more complex with dynamic loading; licensing: struct layouts still derived from headers.

### Option C: Official FICO binding package

Use or contribute to an officially-maintained Rust binding for Xpress.

- **Pros:** Licensing resolved by the binding author; maintenance burden shared.
- **Cons:** No known official Rust bindings exist as of 2026-07; ROML has no relationship with FICO.

### Recommendation

The decision leans toward **Option A (generated bindgen sys crate)**. However, the decision is formally **BLOCKED** pending legal verification with FICO that generating and distributing Rust FFI declarations from Xpress C headers is permitted under the FICO Xpress license. Until resolution:

- `roml-xpress` remains `publish = false`
- Handwritten FFI declarations remain in place, marked `#[allow(dead_code)]` with provenance comments
- No Xpress adapter features are documented as "supported"
- The ROML release train proceeds with `roml` + `roml-highs` only

### 2.2 Current Adapter Architecture

The `XpressAdapter` implements the **legacy** `SolverAdapter` trait with `apply_changes(&[Change])`. It has not been migrated to the frozen `BackendAdapter` trait with `DeltaBatch` operations (see M1.1_CONTRACT_FREEZE.md, Section 3).

Key architectural elements:

- **State tracking:** `col_map` / `row_map` (IndexMap), `var_bounds` / `con_bounds` (HashMap), `integer_vars` / `semicontinuous_vars` (HashSet), `obj_costs` / `obj_senses` (HashMap), `active_obj` (Option).
- **Deletion handling:** Inactive variables are fixed at [0,0]; inactive constraints become non-binding ('N'). Xpress uses dense integer addressing, so deletions require `reindex_after_delete` to keep maps in sync.
- **Bulk-update path:** `apply_bulk_additive_changes` â€” see Section 6 below.
- **Batched row-add optimization:** In `apply_changes`, when a `ConstraintAdded` is followed by its `CoefficientAdded` events, they are combined into a single `XPRSaddrows` call with all coefficients, rather than adding the row empty and calling `XPRSchgcoef` N times.
- **Solve path:** Calls `XPRSlpoptimize` or `XPRSmipoptimize` based on whether integer variables are present. Solution extraction reads LP/MIP status attributes, then calls `XPRSgetlpsol` / `XPRSgetmipsol` / `XPRSgetdblattrib`.
- **Library lifecycle:** `XPRSinit` called exactly once per process via `OnceLock`. `XPRSdestroyprob` on `Drop`.

## 6. Bulk-Update Path Characterization

### 6.1 What the Current Bulk Path Does

The method `apply_bulk_additive_changes` (lines 331â€“538 of `adapter.rs`) is entered when `supports_bulk_additive_sync` returns `true` for a changeset. This predicate excludes any changeset containing:

- `VariableRemoved`
- `VariableActivityChanged`
- `ConstraintRemoved`
- `ConstraintActivityChanged`
- `CoefficientRemoved`
- `CoefficientValueChanged`
- `ObjectiveRemoved`

**Included operations** (the additive set): `VariableAdded`, `VariableBoundsChanged`, `VariableTypeChanged`, `ConstraintAdded`, `ConstraintBoundsChanged`, `CoefficientAdded`, `ActiveObjectiveChanged`.

The bulk path proceeds in three phases:

**Phase 1 â€” Collect new entities (lines 337â€“476):**
- Scans all changes to collect new variables and constraints, merging any subsequent bound/type changes into the collected entities.
- Calls `XPRSaddcols` once for all new variables (with zero coefficients and zero matrix entries).
- If any new variable is integer or binary, calls `XPRSchgcoltype` in a single batch.
- Calls `XPRSaddrows` once for all new constraints (with zero matrix entries).
- Updates `col_map`, `row_map`, `var_bounds`, `con_bounds`, `integer_vars`.

**Phase 2 â€” Apply coefficients and objective changes (lines 478â€“537):**
- Collects constraint-coefficient entries into `XPRSchgmcoef` (bulk coefficient change).
- Applies objective coefficients to cached `obj_costs`.
- If active objective changed, updates `active_obj` and calls `reload_active_objective` which does a bulk `XPRSchgobj` for all columns.

**Phase 3 â€” Fall-through for remaining operations (line 517):**
- Any operation not handled by Phase 1 or Phase 2 (e.g., `ObjectiveAdded`, `ObjectiveSenseChanged`, `ParameterValueChanged`) is dispatched to `apply_one`.

Additionally, the **batched row-add optimization** in `apply_changes` (lines 990â€“1061) groups each `ConstraintAdded` with its immediately following `CoefficientAdded` events for the same constraint, issuing a single `XPRSaddrows` with all coefficients rather than inserting the row empty and adding N coefficients individually.

### 6.2 Operations Batched

| Operation | Phase | Xpress API Call |
|-----------|-------|----------------|
| `VariableAdded` | 1 | `XPRSaddcols` (bulk) |
| `VariableBoundsChanged` | 1 | Merged into new variable declaration |
| `VariableTypeChanged` | 1 | `XPRSchgcoltype` (bulk) |
| `ConstraintAdded` | 1 | `XPRSaddrows` (bulk) |
| `ConstraintBoundsChanged` | 1 | Merged into new constraint declaration |
| `CoefficientAdded` (constraint) | 2 | `XPRSchgmcoef` (bulk) |
| `CoefficientAdded` (objective) | 2 | Cached obj_costs + `XPRSchgobj` via `reload_active_objective` |
| `ActiveObjectiveChanged` | 2 | `reload_active_objective` (bulk zero + bulk load) |

### 6.3 What Must Be Verified Before Migration

When migrating to `DeltaBatch` operations, each bulk behavior must be replicated using typed `ModelOp` variants in a `DeltaBatch`. The following equivalence checks are required:

1. **Phase 1 equivalence (entity addition):** Applying N `AddVariable` ops as separate `apply_one` calls (each `XPRSaddcols` with N=1) must produce identical backend state to a single batch containing N `AddVariable` ops. This is primarily an API-call-count optimization; semantics must be identical.

2. **Phase 2 equivalence (bulk coefficients):** Applying N `SetCell` ops as N individual `XPRSchgcoef` calls must produce identical constraint matrix state to a single batch with N `SetCell` ops using `XPRSchgmcoef`. Edge cases: rows/columns that were just added in Phase 1, overlapping coefficient assignments for the same (row,col).

3. **Batched row-add equivalence:** Applying `ConstraintAdded` + N `SetCell` as individual operations must produce identical state to the combined `XPRSaddrows` with coefficients. Edge cases: partial coefficient sets (not all cells in the new row are set in the same batch).

4. **Objective bulk reload equivalence:** Applying ob1 + ob2 + ob3 via `XPRSchgobj` individually must match a single `reload_active_objective` call. Edge cases: zero-column state (ncols=0), column count changes between reloads.

5. **Non-additive fallback correctness:** The `apply_one` path for operations excluded from the bulk path (`CoefficientRemoved`, `CoefficientValueChanged`, `VariableRemoved`, etc.) must produce identical state whether applied individually or interleaved with bulk operations.

6. **Deletion+reindex correctness:** Operations that delete columns/rows must be tested after bulk additions to verify `reindex_after_delete` handles all gap-closing scenarios correctly. The current `IndexMap` logic decrements every stored index > deleted by 1; this must be verified when bulk-additions precede deletions.

### 6.4 Proposed Migration Strategy

1. **Implement `BackendAdapter` trait** for `XpressAdapter`, providing `apply_batch` that processes `&DeltaBatch` (a list of `ModelOp`).
2. **Write a scalar-bulk comparator** that, for a given model mutation sequence, applies it via (a) individual `apply_one` calls and (b) bulk-optimized batching, then asserts `normalized_view` equivalence at each revision.
3. **Replace `apply_bulk_additive_changes`** with the new `DeltaBatch`-based `apply_batch`, preserving the bulk API calls as performance optimizations (same number of XPRS C calls, same semantics, same state invariants).
4. **Run the scalar-bulk comparator** on the full `differential_harness` random mutation sequences (seeded, deterministic) to verify equivalence across all operation combinations.
5. **Delete `apply_bulk_additive_changes`** and the batched row-add optimization once equivalence is proven â€” the optimization logic lives inside the new `apply_batch` implementation.
