## 1. Protocol Types (frozen)

### 1.1 Revision and Delta

```
ModelRevision(u64)                    — monotonic, overflow-safe
DeltaBatch { from, to, operations }   — immutable, ordered, self-contained
ModelOp                               — 17 typed variants, no adjacent-event dependency
Journal                               — BTreeMap<Revision, DeltaBatch>, sequential-only
```

**Laws:**
- `from < to` for every committed batch
- Journal rejects gaps: `record(batch)` fails if `batch.from != journal.latest`
- `deltas_since(r)` returns all batches `>= r` in revision order
- Operations are deterministic and self-contained (adapters need no model access)

### 1.2 Snapshot and Projection

```
ModelSnapshot                         — deterministic projection of canonical state at a revision
take_snapshot(revision, entities)     — produces identical snapshots for identical state
```

**Laws:**
- Snapshots sorted by entity ID for deterministic comparison
- `project(snapshot r1) == apply(project(snapshot r0), deltas r0→r1)` (commuting square)

### 1.3 Cursor and Health

```
AdapterCursor { applied_revision, health }
AdapterHealth ::= Ready | RequiresRebuild | Terminal
```

**Laws:**
- Cursor advances only after full batch acknowledgement
- `advance(batch)` fails if `batch.from != cursor.applied_revision`
- Health transitions: `Ready → RequiresRebuild → Ready` (after rebuild) or `→ Terminal`
- Terminal adapters must be recreated

### 1.4 Apply Outcomes

```
ApplyOutcome ::=
    Applied { new_revision }
  | RequiresRebuild { failed_at_op, reason }
  | RecoverableFailure { reason }
  | DirtyFailure { reason }
```

**Laws:**
- `Applied`: all operations succeeded, cursor advanced, backend state matches revision
- `RequiresRebuild`: operation unsupported incrementally; backend state unchanged; caller rebuilds from snapshot
- `RecoverableFailure`: operation failed; backend state unchanged; delta preserved for retry
- `DirtyFailure`: partial mutation occurred; backend must be rebuilt from snapshot

### 1.5 Capability Algebra

```
BackendCapabilities {
    add_variable, add_constraint, set_coefficient, set_bounds,
    set_objective, delete, lp, mip, callbacks, solution,
    duals, reduced_costs, semicontinuous, semiinteger, parameter_update
}
```

**Laws:**
- Every capability defaults to `false`
- Backend declares capabilities at construction; they are immutable
- `validate_request(request, capabilities) → Vec<Rejection>`
- Unsupported operations produce `RequiresRebuild`, never silent degradation

### 1.6 Solve Request and Effective Configuration

```
SolveRequest {
    lp_algorithm, time_limit_secs, mip_rel_gap, mip_abs_gap,
    threads, enable_output, random_seed, extra_options
}

SolveResult {
    effective_configuration, termination, solution
}

EffectiveConfig {
    lp_algorithm, time_limit_secs, mip_rel_gap, threads,
    enable_output, adjustments: Vec<ConfigAdjustment>,
    rejections: Vec<ConfigRejection>
}
```

**Laws:**
- Every option is either applied, adjusted (with reason), or rejected (with reason)
- "Unsupported options are silently ignored" is forbidden
- `effective_configuration` is always populated with what was actually used
- Request is consumed by one solve attempt; failed attempts don't consume policy

### 1.7 Error Classification

```
BackendError {
    message, category: ErrorCategory, native_code, health_effect: HealthEffect
}

ErrorCategory ::=
    InvalidInput | Unsupported | LibraryNotFound | LicenseFailure |
    Numerical | OutOfMemory | Internal | Limit | Unknown

HealthEffect ::= None | Recoverable | RequiresRebuild | Terminal
```

**Laws:**
- Every native return code is checked; no ignored errors
- `LibraryNotFound` → `Terminal` (cannot proceed without library)
- `Unsupported` → `RequiresRebuild` (rebuild may bypass the unsupported path)
- `Numerical` → `Recoverable` (retry may succeed)

### 1.8 Status Lattice

```
TerminationStatus ::=
    Optimal | Feasible | Infeasible | Unbounded |
    TimeLimit | IterationLimit | NodeLimit |
    Interrupted | NumericalIssue | Error | Unknown
```

**Information ordering** (top = most informative):
```
Optimal > Feasible > {Infeasible, Unbounded} > {TimeLimit, IterationLimit, NodeLimit} > Interrupted > NumericalIssue > Error > Unknown
```

**Laws:**
- `Optimal` implies a feasible solution with proven optimality
- `Feasible` implies a feasible solution without optimality proof (MIP incumbent)
- `Infeasible` and `Unbounded` are proof states with no solution
- Limit statuses may or may not have a feasible solution
- `Error` and `Unknown` guarantee no usable solution

### 1.9 Solution Views

```
SolveSolution {
    variable_values: Vec<(VarId, f64)>,
    objective_value: Option<f64>,
    dual_values: Option<Vec<(ConId, f64)>>,
    reduced_costs: Option<Vec<(VarId, f64)>>,
}
```

**Laws:**
- `variable_values` is populated when a feasible solution exists
- `dual_values` is `Some` only when `BackendCapabilities.duals` is true AND the solver produced duals
- `reduced_costs` is `Some` only when `BackendCapabilities.reduced_costs` is true
- Solution extraction never panics on missing data; absent fields are `None`

### 1.10 Callback Taxonomy

Callbacks are solver notifications during solve. ROML does not guarantee uniform callback semantics across backends.

```
CallbackCapability ::=
    Progress         — observe iteration/log messages
  | Interrupt        — request early termination
  | Incumbent        — observe new incumbent solutions (MIP)
  | LazyConstraints  — add constraints during solve (MIP, only where officially supported)
  | UserCuts         — add cutting planes during solve (MIP, only where officially supported)
```

**Laws:**
- Every backend declares which callback capabilities it supports
- Unsupported callback capabilities are rejected at adapter construction, never silently ignored
- Callback handlers must not panic across FFI boundaries
- MOSEK: task mutation inside callbacks is prohibited by official documentation. ROML must not offer lazy-constraint/user-cut callbacks for MOSEK unless an official out-of-callback protocol is proven.
- HiGHS: callback support depends on pinned `highs-sys` API surface. Unsupported callback types produce `UnsupportedCapability` error.
- Xpress: callback support depends on binding decision (M1.7).

## 3. Backend Adapter Trait (frozen sketch)

The existing `SolverAdapter` trait will be replaced. The frozen contract is:

```rust
trait BackendAdapter {
    /// Backend identity and capabilities.
    fn info(&self) -> &BackendInfo;

    /// Apply a delta batch. Returns outcome; never leaves backend
    /// in unobservable partially-mutated state on error.
    fn apply_batch(&mut self, batch: &DeltaBatch) -> Result<ApplyOutcome, BackendError>;

    /// Rebuild from a canonical snapshot. Must produce state identical
    /// to applying all deltas from r0 to snapshot revision.
    fn rebuild(&mut self, snapshot: &ModelSnapshot) -> Result<(), BackendError>;

    /// Solve with the given request. Returns result with effective
    /// configuration. Never silently ignores unsupported options.
    fn solve(&mut self, request: &SolveRequest) -> Result<SolveResult, BackendError>;

    /// Extract the current solution view (may be None if no solution).
    fn solution(&self) -> Option<&SolveSolution>;

    /// Get the adapter's synchronization cursor.
    fn cursor(&self) -> &AdapterCursor;

    /// Get mutable cursor (for sync coordinator use).
    fn cursor_mut(&mut self) -> &mut AdapterCursor;
}
```

Implementation must wait for M1.2 (HiGHS) and beyond. The existing `SolverAdapter` trait remains until the new contract is implemented.
