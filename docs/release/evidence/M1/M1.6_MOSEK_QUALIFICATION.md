# M1.6 — MOSEK Backend Qualification Plan

**Date:** 2026-07-14
**Status:** PENDING — not yet qualified; plan defines gate conditions
**Decision owner:** sk-surya
**Dependencies:** M1.1 (contract freeze), M1.2 (HiGHS reference backend), M1.3 (differential harness)

---

## 1. Overview

MOSEK is a commercial linear and mixed-integer optimization solver from MOSEK
ApS (Denmark). ROML includes a skeleton MOSEK backend at `roml-mosek/` that
is not yet qualified for release. This document audits the current state,
defines the migration path to the official `mosek` Rust crate, and establishes
gate conditions for graduation from `publish = false`.

## 2. Current State Audit

### 2.1 Directory structure

```
roml-mosek/
  Cargo.toml               — publish = false; no workspace dependencies on mosek crate
  build.rs                 — handwritten MOSEK library discovery via MOSEK_BINDIR / MOSEK_ROOT
  src/
    lib.rs                 — re-exports adapter types
    ffi.rs                 — handwritten FFI declarations for ~30 MOSEK C functions
    adapter.rs             — MosekAdapter impl (~1060 lines)
    index_map.rs           — VarId/ConId <-> MOSEK dense-index mapping
  tests/
    integration.rs         — 8 integration tests (LP, MIP, bound changes, infeasibility)
  MOSEK_API_CHEATSHEET.md  — internal developer reference
```

### 2.2 Handwritten FFI (src/ffi.rs)

The file at `roml-mosek/src/ffi.rs` contains manually transcribed C function
declarations, struct layouts, and constant definitions derived from the MOSEK
C API headers. Key characteristics:

- **Opaque handle types** (`MosekEnv`, `MosekTask`) typed as `*mut c_void`
- **~30 extern "C" function declarations** covering environment/task lifecycle,
  parameters, variable/constraint CRUD, optimization, and solution extraction
- **~40 constant definitions** for bound keys, solution types, optimizer types,
  callback codes, and info array indices — all hand-copied with values that
  match the C enum constants
- **No automated derivation** — the FFI is entirely manual, without any
  `bindgen`, version-pinning, or CI-validator to catch API drift

**Audit finding:** This is a known-worst-practice pattern. Every MOSEK version
could change enum values, add/remove fields, or alter ABI. The same concern
documented in `XPRESS_BINDING_DECISION.md` (header-derivative redistribution,
version fragility) applies here.

### 2.3 Callback Task-Mutation Strategy (critical finding)

The `mosek_callback_trampoline` function in `adapter.rs` (lines 636–768)
registers a C callback via `MSK_putcallbackfunc` and **mutates the MOSEK
task from inside the callback** — calling `MSK_appendcons` and
`MSK_putaijlist` to add lazy constraints / user cuts while `MSK_optimize`
is still running.

**This violates MOSEK's official documentation.** The MOSEK C API explicitly
states that the task is not fully re-entrant during a callback — calling
model-modification functions (`MSK_appendcons`, `MSK_putaij`, etc.) from
inside a callback is unsupported and may produce undefined behavior.

The cheatsheet at `MOSEK_API_CHEATSHEET.md` acknowledges this workaround:
the callback must terminate the solve (return non-zero), and the adapter
must re-call `MSK_optimize` in a loop. The current implementation **does
not follow this pattern correctly** — it both modifies the task inside the
callback AND returns a non-zero value to terminate. The loop re-solve is
present, but the task mutation inside the callback remains a violation of
MOSEK's stated contract.

**Impact:** This is a blocking defect. The callback strategy must be
redesigned before qualification.

### 2.4 Build System (build.rs)

`build.rs` uses a two-variable heuristic to locate the MOSEK shared library:

1. `MOSEK_BINDIR` — direct path to the directory containing `libmosek64`
2. `MOSEK_ROOT` — installation root; guesses the platform subdirectory

This differs from the official `mosek` Rust crate, which uses the standard
`MOSEK_INST_BASE` environment variable established by MOSEK's own installer
and documentation.

**Audit finding:** Custom path-guessing duplicates (and may conflict with)
the official crate's discovery mechanism. Using `MOSEK_INST_BASE` is the
documented MOSEK convention and should be preferred.

### 2.5 Error Handling

The adapter currently:
- Converts all non-zero return codes to `SolverError::InternalError` via a
  `check()` helper (line 39–45 of adapter.rs)
- Does not distinguish between `LibraryNotFound`, `LicenseFailure`, and
  `Numerical` error categories as required by the M1.1 frozen contract
- Does not map MOSEK return codes to `BackendError` types
- Does not implement `HealthEffect` transitions

### 2.6 License Handling

The current build system does not attempt to create a MOSEK environment
until runtime. License failures surface as opaque `MSK_makeenv` return codes
(assertion at line 185: `assert!(ret == ffi::RES_OK, ...)`). There is no
distinct code path for "no license file found" vs "library not installed"
vs "library loaded but license expired."

### 2.7 Test Coverage

`tests/integration.rs` contains 8 tests exercising basic solve paths:
- Simple LP solve
- Incremental bound change
- Constraint add/remove
- Parameter coefficient update
- Binary MIP
- Objective switch
- Infeasible model detection
- Solution enrichment (objective value, duals, reduced costs)

These tests provide basic smoke-testing but do **not** constitute the
full backend conformance suite (46 tests in `backend_contract.rs`) or
the differential harness (23 tests in `differential_harness.rs`).

## 3. Migration Plan

### 3.1 Target: Official `mosek` Rust Crate (v11.2.0)

The official `mosek` crate (version 11.2.0 at time of writing) is published
on crates.io as the canonical Rust API for MOSEK optimization tools. It
provides:

- Type-safe Rust bindings for all MOSEK C API functions
- Safe wrapper types for `Env` and `Task`
- Full solution extraction, parameter management, and callback support
- Standard `MOSEK_INST_BASE` environment variable for library discovery
- Version-locked against the MOSEK 11 API surface

**Decision:** Replace all handwritten FFI in `roml-mosek/src/ffi.rs` with
the official `mosek` crate. This mirrors the M1.1 contract requirement for
version-safe binding strategies and avoids the header-derivative licensing
concerns identified in `XPRESS_BINDING_DECISION.md`.

### 3.2 Step-by-Step Migration

#### Step 1: Dependency Switch

1. Add `mosek = "11.2"` to `roml-mosek/Cargo.toml` dependencies
2. Remove `roml-mosek/src/ffi.rs` entirely
3. Remove `roml-mosek/build.rs` entirely (library discovery handled by crate)
4. Update `roml-mosek/src/lib.rs` to remove the `mod ffi` and `mod index_map`
   declarations

**Verification:** `cargo check --package roml-mosek` must compile without
errors and without any handwritten FFI files.

#### Step 2: Adapter Rewrite — Model Building

Rewrite `roml-mosek/src/adapter.rs` to use the official crate's safe types:

- Replace `MosekEnv` / `MosekTask` with `mosek::Env` / `mosek::Task`
- Replace all `unsafe { ffi::MSK_*() }` calls with crate method calls
- Remove `index_map.rs`: the official crate's `Task` type manages indices

The apply-loop (17 `ModelOp` variants) must be reimplemented against the
official crate's API surface. Each operation maps to one or more `Task`
method calls, preserving the existing semantic behavior.

**Verification:** All existing 8 integration tests pass.

#### Step 3: Callback Redesign (critical)

**Replace the current callback-in-callback mutation with the legally supported
collect-terminate-apply-reoptimize pattern:**

Before qualification, the current design (`mosek_callback_trampoline` adding
constraints during `MSK_optimize`) **must be removed**. The replacement
strategy depends on the callback capability:

##### For Progress / Interrupt callbacks (officially supported):

MOSEK supports these callback types natively. The callback can read solution
data via `MSK_getxx` and return 0 (continue) or non-zero (terminate). No
task mutation is required inside the callback. This maps to
`CallbackCapability::Progress` and `CallbackCapability::Interrupt`.

Implementation:
```
C callback fires with CALLBACK_IM_MIO or CALLBACK_HEARTBEAT
  -> reads variable values, objective bounds from douinf/intinf arrays
  -> invokes user CallbackHandler with CallbackData (read-only)
  -> handler returns Accept (continue) or Interrupt (terminate)
  -> callback returns 0 (continue) or non-zero (terminate with RES_TRM_USER_CALLBACK)
```

##### For Incumbent callbacks (officially supported via read-only):

MOSEK fires `CALLBACK_NEW_INT_MIO` when a new integer-feasible solution is
found. The callback can read the solution via `MSK_getxx` but **must not**
modify the task. This maps to `CallbackCapability::Incumbent`.

Implementation:
```
C callback fires with CALLBACK_NEW_INT_MIO
  -> reads incumbent solution (MSK_getxx with SOL_ITG)
  -> reads primal/dual bounds from douinf
  -> invokes user CallbackHandler with CallbackData (read-only)
  -> handler returns Accept or Reject
  -> callback returns 0 (continue; accepted) or non-zero (terminate; rejected)
```

If the user handler wishes to add cuts after a rejected incumbent, the pattern is:

```
optimize loop:
  1. Call MSK_optimize
  2. Extract solution
  3. If callback was NOT installed:
       return solution normally
  4. If callback terminated solve (RES_TRM_USER_CALLBACK):
       - collect any pending cuts from the handler
       - apply cuts outside the callback (using Task methods)
       - if cuts were added: goto 1 (re-optimize)
  4. If MSK_optimize completed normally:
       return solution
```

**This is the only safe pattern for lazy constraints / user cuts with MOSEK.**
It must be verified against the MOSEK API documentation before implementation.

##### For Lazy Constraints / User Cuts (blocked — see risk register):

MOSEK does not have an official API for adding constraints from a callback
that auto-continues optimization. The collect-terminate-apply-reoptimize
pattern described above is the closest legal approximation, but it has
limitations:

- Each re-optimize cycle restarts from scratch or from the last solution
  (MOSEK does not support warm-start after task modification)
- Convergence guarantees are user-dependent
- Performance degrades with many cut cycles

**REML must declare `CallbackCapability::LazyConstraints` and
`CallbackCapability::UserCuts` as UNSUPPORTED for the MOSEK backend
until MOSEK adds official support for these patterns.**

#### Step 4: Environment Variable Standardization

Replace `MOSEK_BINDIR` / `MOSEK_ROOT` detection with `MOSEK_INST_BASE`:

- The official `mosek` crate respects `MOSEK_INST_BASE` for library discovery
- After migration, `roml-mosek/build.rs` is removed entirely
- If the official crate does not support dynamic-link-time-only resolution,
  fall back to the official crate's own `links` mechanism
- Document `MOSEK_INST_BASE` as the sole environment variable for MOSEK
  library path configuration

**Verification:** Running tests with only `MOSEK_INST_BASE` set (and without
`MOSEK_BINDIR` or `MOSEK_ROOT`) must succeed.

#### Step 5: Error Classification

Implement the M1.1 error classification contract:

| MOSEK Return Code Range | ErrorCategory | HealthEffect |
|------------------------|---------------|--------------|
| RES_OK (0)             | — (success)   | —            |
| RES_TRM_MAX_TIME, etc. | Limit         | Recoverable   |
| RES_TRM_USER_CALLBACK  | — (user stop) | —            |
| RES_ERR_LICENSE        | LicenseFailure| Terminal     |
| Library load failure   | LibraryNotFound| Terminal    |
| RES_ERR_NUMCON, etc.   | InvalidInput  | Recoverable   |
| RES_ERR_INVINTERTERM   | Numerical     | Recoverable   |
| RES_ERR_SPACE          | OutOfMemory   | Terminal      |
| RES_ERR_INTERNAL       | Internal      | Terminal      |
| All others             | Unknown       | Depends       |

Implementation steps:
1. Map MOSEK `rescodee` to `ErrorCategory` using a `match` on documented codes
2. Populate `BackendError { message, category, native_code, health_effect }`
3. Set `health_effect` based on error category per M1.1 contract
4. Remove the generic `check()` / `mosek_err()` fallback

#### Step 6: Conformance Harness Integration

1. Enable MOSEK-specific test runs against `tests/backend_contract.rs` (46 tests)
2. Enable MOSEK-specific test runs against `tests/differential_harness.rs` (23 tests)
3. All 46 + 23 tests must pass to satisfy the M1.1 contract freeze

This will likely require:
- A `tests/backends.rs` module or feature-gated test configuration
- CI matrix entry for MOSEK (see Gating Condition 3)
- Resolution of any backend-specific assertion failures in the generic harness

#### Step 7: Documentation and Feature Gates

- Update `roml-mosek/README.md` with official crate dependency information
- Document the supported callback capabilities explicitly
- Add `feature = "mosek"` gate on any MOSEK-specific tests
- Document `MOSEK_INST_BASE` usage
- Document the known callback limitations

## 4. Gate Conditions

The following conditions must be **true** before MOSEK can graduate from
`publish = false`:

### Condition 1: Official Binding Migration Complete

- [ ] `roml-mosek/Cargo.toml` depends on `mosek = "11.2"` (not handwritten FFI)
- [ ] `roml-mosek/src/ffi.rs` has been removed
- [ ] `roml-mosek/build.rs` has been removed
- [ ] `roml-mosek/src/adapter.rs` uses only official crate types (`mosek::Env`,
      `mosek::Task`, etc.)
- [ ] `roml-mosek/src/index_map.rs` has been removed or made redundant

### Condition 2: Callback Strategy Approved per MOSEK Documentation

- [ ] The current callback task-mutation pattern has been removed
- [ ] A review against the MOSEK C API documentation has been completed
- [ ] Supported callback types are explicitly declared and match MOSEK's
      documented safe subset
- [ ] Lazy constraints and user cuts are explicitly marked as UNSUPPORTED
      for the MOSEK backend
- [ ] The collect-terminate-apply-reoptimize pattern (if implemented) has
      been reviewed for correctness against MOSEK's stated contract

### Condition 3: Licensed CI Runner Available

- [ ] A CI runner with a valid MOSEK license is configured
- [ ] The runner can run `roml-mosek` test suite without credential exposure
- [ ] The CI build does not fail when MOSEK is absent (graceful feature gating)
- [ ] A non-MOSEK CI pipeline runs for non-MOSEK changes

### Condition 4: Backend Contract Tests Pass

- [ ] `tests/backend_contract.rs` — all 46 tests pass against the MOSEK backend
- [ ] All 17 `ModelOp` variants are exercised and verified
- [ ] Snapshot identity, rebuilding, commuting square, and cascading deletion
      are confirmed for MOSEK

### Condition 5: Differential Harness Tests Pass

- [ ] `tests/differential_harness.rs` — all 23 tests pass against the MOSEK backend
- [ ] Fault injection at every apply sub-step works correctly
- [ ] Multi-cursor independence is verified
- [ ] Solve request negotiation round-trips correctly
- [ ] Status mapping for every `TerminationStatus` variant is verified

### Condition 6: Error Classification Implemented

- [ ] MOSEK return codes map to `BackendError` with correct `ErrorCategory`
- [ ] Library-not-found and license-failure are distinct, with correct
      `HealthEffect::Terminal`
- [ ] All native return codes are checked; no ignored errors

### Condition 7: Documentation Updated

- [ ] Supported callback capabilities declared in `BackendCapabilities`
- [ ] `MOSEK_INST_BASE` documented as the library discovery mechanism
- [ ] Known limitations (no lazy constraints, no user cuts) documented
- [ ] License requirements documented

## 5. Non-Blocking Status

**MOSEK qualification does NOT block the core/ HiGHS release train.**

The ROML workspace is structured so that `roml` (core crate), `roml-highs`,
`roml-xpress`, and `roml-mosek` are independent workspace members with
`publish = false` on the commercial backends. The M1 release train proceeds
on the following schedule:

| Milestone | Blocked by MOSEK? | Release target |
|-----------|-------------------|----------------|
| M1.1 (Contract Freeze)      | No  | Complete |
| M1.2 (HiGHS Adapter)        | No  | Next     |
| M1.3 (Differential Harness) | No  | Next     |
| M1.4 (Error Classification) | No  | Next     |
| M1.5 (Release)              | No  | HiGHS only |
| **M1.6 (MOSEK Qualification)** | — | Post-M1.5 |

Key points:
- `roml-mosek/Cargo.toml` sets `publish = false` and will remain so until
  all gate conditions are satisfied
- The `roml` crate itself has zero dependency on MOSEK
- Test infrastructure (`backend_contract.rs`, `differential_harness.rs`) is
  backend-agnostic and does not require MOSEK to pass
- No ROML user requires MOSEK; it is an optional commercial backend
- MOSEK qualification can proceed asynchronously on a separate branch or
  workstream

## 6. Risk Register

### Risk 1: Callback Redesign Complexity

| Attribute | Detail |
|-----------|--------|
| **Description** | The collect-terminate-apply-reoptimize pattern for lazy constraints is fragile. Each re-optimize discards the MOSEK branch-and-bound tree, restarting from scratch. For models requiring many cut cycles, performance degrades polynomially. Both the MOSEK API documentation and the internal cheatsheet confirm this limitation. |
| **Likelihood** | High — the current callback code must change regardless, and the legal replacement introduces performance unknowns. |
| **Impact** | Medium — progress/incumbent callbacks are straightforward; lazy constraints may be practically unusable even with the correct pattern. |
| **Mitigation** | Declare lazy constraints / user cuts as UNSUPPORTED for MOSEK. Focus on progress, interrupt, and incumbent callbacks which have clean, documented semantics. |
| **Owner** | sk-surya |

### Risk 2: License Availability for CI

| Attribute | Detail |
|-----------|--------|
| **Description** | MOSEK requires a commercial license. CI runners must have access to `mosek.lic`. Without a license, all MOSEK tests will fail at the environment-creation step (RES_ERR_LICENSE). Licensing for open-source CI services is typically not provided by MOSEK ApS. |
| **Likelihood** | High — no MOSEK license is currently provisioned for CI. |
| **Impact** | High — without licensed CI, no MOSEK tests can run in CI, and MOSEK cannot graduate past `publish = false`. |
| **Mitigation** | (a) Provision a self-hosted runner with a MOSEK Personal Academic License (free for academics). (b) Feature-gate MOSEK tests so the main CI pipeline does not fail when MOSEK is absent. (c) Run MOSEK CI only on merge to a release-candidate branch, not on every commit. |
| **Owner** | sk-surya |

### Risk 3: Official Crate API Surface Coverage

| Attribute | Detail |
|-----------|--------|
| **Description** | The official `mosek` Rust crate may not expose every MOSEK C API function that ROML needs. Specifically: callback function registration, solution enrichment (duals, reduced costs), callback data arrays (douinf/intinf), and semi-continuous variable types must be verified against the crate's API. If the crate is missing critical functionality, ROML must contribute upstream or fall back to a hybrid approach. |
| **Likelihood** | Medium — the MOSEK crate is mature (v11.2.0) but usage patterns vary. |
| **Impact** | Medium — could require upstream contribution, slowing the migration timeline. |
| **Mitigation** | Conduct an API coverage audit against the functions currently used in `adapter.rs`. Patch the official crate if gaps exist (MOSEK's crate repository is public). |
| **Owner** | sk-surya |

### Risk 4: Thread-Safety Requirements

| Attribute | Detail |
|-----------|--------|
| **Description** | MOSEK's C API has specific thread-safety rules: environments are not shareable across threads; tasks are bound to the creating thread. ROML's `BackendAdapter` trait requires `Send + Sync` compatibility for multi-cursor and parallel-solve scenarios. The current implementation uses `unsafe impl Send for MosekAdapter {}` at line 175 of adapter.rs. |
| **Likelihood** | Medium — multi-cursor tests (differential harness) will validate this. |
| **Impact** | High — if MOSEK cannot provide the required thread-safety guarantees, the MOSEK adapter may need to use a Mutex-protected single-writer pattern. |
| **Mitigation** | (a) Verify MOSEK thread-safety documentation. (b) If MOSEK is only `Send` (not `Sync`), restrict the adapter accordingly. (c) Use a connection-pool pattern if task creation per thread is required. |
| **Owner** | sk-surya |

### Risk 5: Commuting Square Violations

| Attribute | Detail |
|-----------|--------|
| **Description** | The M1.1 contract requires that `project(snapshot r1) == apply(project(snapshot r0), deltas r0->r1)` — the commuting square. MOSEK's dense-index addressing (variables/constraints shift after deletions) creates subtle ordering effects. The `reindex_after_delete` logic in `index_map.rs` attempts to handle this, but errors here would cause differential harness failures. |
| **Likelihood** | Medium — deletion indices are a known source of bugs in backend adapters. |
| **Impact** | Medium — gate condition 5 (differential harness) catches these. |
| **Mitigation** | The existing 23 differential harness tests exist specifically for this purpose. Gate on 100% pass rate. |
| **Owner** | sk-surya |

### Risk 6: Migration Effort and Schedule

| Attribute | Detail |
|-----------|--------|
| **Description** | The adapter rewrite (Steps 2–5) touches ~1100 lines of code in `adapter.rs`. Each of the 17 `ModelOp` variants must be re-verified against the official crate's API. Callback redesign introduces architectural changes to the solve method. |
| **Likelihood** | Medium — scope is well-defined but non-trivial. |
| **Impact** | Medium — delays MOSEK qualification but does not affect the main release. |
| **Mitigation** | (a) Phased implementation: model-building first, solutions second, callbacks last. (b) Early API coverage audit (Risk 3) to surface crate gaps. (c) Parallel workstream if resource permits. |
| **Owner** | sk-surya |

## 7. Qualification Exit Criteria

The MOSEK backend is considered qualified when:

1. All 7 gate conditions (Section 4) are satisfied
2. The risk register (Section 6) is reviewed and accepted by the decision owner
3. `roml-mosek/Cargo.toml` has `publish = false` removed (or changed to
   `publish = true` with appropriate documentation)
4. A MOSEK-specific chapter is added to the release documentation covering:
   - Installation requirements (`MOSEK_INST_BASE`, license file)
   - Supported capabilities (and explicit list of unsupported capabilities)
   - Callback limitations (no lazy constraints, no user cuts)
   - Known version compatibility (MOSEK 11.x only)
5. CI pipeline includes a MOSEK-licensed runner in the build matrix
