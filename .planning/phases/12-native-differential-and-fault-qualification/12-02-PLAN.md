    2. Port `FaultOutcome` enum: `Recoverable(String)`, `Dirty(String)` — unchanged from worktree.

    3. Port `FaultConfig` struct: `fail_at_op: usize`, `outcome: FaultOutcome` — unchanged.

    4. Port `FaultInjectingBackend` struct: replaces `inner: ReferenceBackend` with `inner: Box<dyn BackendSession>`, keeps `global_op_count: usize` and `fault: Option<FaultConfig>`. Constructor `new(session: Box<dyn BackendSession>)`.

    5. Port `configure_fault` and `clear_fault` methods — unchanged logic.

    6. Port `apply_batch` method (the core adaptation): instead of calling `self.inner.apply_op(op)` directly (ReferenceBackend API), the FaultInjectingBackend must MIMIC the fine-grained apply pattern. Since BackendSession only exposes `synchronize(DeltaBatch)`, the FaultInjectingBackend needs to:
       - Keep a `DeltaBatch` accumulator during fault injection
       - On RecoverableFailure: do NOT apply the op, return RecoverableFailure
       - On DirtyFailure: the op IS applied to `self.inner` via synchronize, then DirtyFailure is returned

       The key change: For the DirtyFailure case, apply the current op via `self.inner.synchronize(Synchronization::DeltaBatch(single_op_batch))` and return DirtyFailure AFTER the op has been applied. Keep a throwaway cursor for this purpose.

       For RecoverableFailure: simply do NOT apply the op and return RecoverableFailure (state unchanged). No accumulative synchronize needed.

       For non-faulted ops in the dirty path: batch multiple ops and call synchronize once.

    7. Port `rebuild` method: delegates to `self.inner.synchronize(Synchronization::Rebuild(snapshot.clone()))`.

    8. Port `normalized_view` method: Comment that this is only available for ReferenceBackend-implementations. When `Box<dyn BackendSession>` is the inner type, access via downcast: `self.inner.deref().as_any().downcast_ref::<ReferenceBackend>().map(|rb| rb.normalized_view())`. This requires adding `fn as_any(&self) -> &dyn Any` to the BackendSession trait OR keeping the test ReferenceBackend-specific. Given the condition, keep it as: the FaultInjectingBackend in this test file is used ONLY with `inner: ReferenceBackend` via `RefBackendFixture`, and normalized_view access happens via a separate reference. Keep a `impl Deref<Target = ReferenceBackend>` on FaultInjectingBackend when inner is ReferenceBackend — or simpler: add a `fn inner_ref(&self) -> &ReferenceBackend` that panics if inner is not ReferenceBackend. Since these tests are ReferenceBackend-specific (fault tests compare state), this is acceptable.

       Simpler design: Parameterize FaultInjectingBackend on the concrete backend type:
       ```
       struct FaultInjectingBackend<T: BackendSession> {
           inner: T,
           ...
       }
       ```
       Then `normalized_view()` is available when T: AsRef<ReferenceBackend> or via a bound. For the test, use `FaultInjectingBackend<ReferenceBackend>`.

       Actually the simplest approach: keep FaultInjectingBackend generic over `T: BackendSession` and add `fn normalized_view(&self) -> Option<NormalizedView>` that returns None for non-ReferenceBackend types. The tests `dx_fault_injection_*` will use `FaultInjectingBackend<ReferenceBackend>` and access normalized_view via `self.inner.normalized_view()`.

    9. Port the three fault injection tests from the worktree (lines 1513-1763):
       - `dx_fault_injection_recoverable_failure`: verifies RecoverableFailure keeps state unchanged
       - `dx_fault_injection_dirty_failure`: verifies DirtyFailure partially applies ops, cursor NOT advanced
       - `dx_fault_injection_rebuild_recovery`: verifies rebuild after DirtyFailure produces correct state

    Add id helper functions (`var_id`, `con_id`, `obj_id`, `param_id`) and the `fault`-related test module markers at the top of the file.
