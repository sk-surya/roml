    **C1: Empty model test**
    - Create session.
    - Build empty ModelSnapshot: no variables, constraints, objectives, cells, or parameters.
    - Call `session.synchronize(Synchronization::Rebuild(snapshot))`.
    - Assert SyncReceipt has AdapterHealth::Ready.
    - Create SolveRequest::default().
    - Call `session.solve(&SolveRequest::new())`.
    - Assert result is Ok, status is TerminationStatus::Optimal (empty model is trivially optimal per AD-6).

    **C2: Full rebuild test (all entity types)**
    - Build a ModelSnapshot with:
      - 3 variables: continuous, integer, binary (various bounds, some inactive).
      - 2 constraints: one with upper bound, one with lower bound.
      - 1 objective (maximize) with cells on all 3 variables.
      - Cells connecting some variables to constraints.
    - Rebuild from snapshot.
    - Assert SyncReceipt.health == Ready.
    - Solve.
    - Assert optimal with expected objective direction.
    - Extract solution and assert values are reasonable.

    **C3: Incremental delta test (all 16 ModelOp variants)**
    - Start with empty session (rebuild empty ModelSnapshot -> ready).
    - Apply a DeltaBatch with each of the 16 ModelOp variants individually:
      - AddVariable (continuous, bounds non-negative).
      - SetVariableBounds (tighten bounds).
      - SetVariableActive (deactivate, then reactivate).
      - SetVariableType (change to integer).
      - AddConstraint (upper bound).
      - SetConstraintBounds (change to lower bound).
      - SetConstraintActive (deactivate, then reactivate).
      - SetCell (add coefficient).
      - RemoveCell (remove coefficient).
      - AddObjective (maximize).
      - SetObjectiveCell (add objective coefficient with constant).
      - SetActiveObjective (switch to this objective).
      - RemoveObjective.
      - RemoveVariable.
      - RemoveConstraint.
      - SetParameter (set a parameter value).
    - For RemoveVariable and RemoveConstraint ordering: ensure the entity exists before removing.
    - After each delta batch, assert the session is Ready and revision advances.
    - Solve after setup, assert solution quality.

    **C4: Commuting square test** (snapshot(r1) == apply(snapshot(r0), deltas r0->r1))
    - Build a ModelSnapshot at r0.
    - Rebuild session A from r0.
    - Apply DeltaBatch(r0->r1) to session A.
    - Take ModelSnapshot at r1 (or build a second snapshot representing r1).
    - Rebuild session B from r1.
    - Solve both sessions with the same SolveRequest.
    - Assert solutions are equivalent (objective values match within epsilon).
    - This proves rebuild equivalence per M1R-C5.

    **C5: Activity toggling** (deactivate/reactivate preserves bounds)
    - Add a variable with specific bounds [1.0, 10.0].
    - Solve (should get x >= 1).
    - Deactivate variable (SetVariableActive { active: false }).
    - Solve (variable should be fixed at 0, effectively absent).
    - Reactivate variable (SetVariableActive { active: true }).
    - Solve (bounds restored, same result as first solve).
    - Prove bounds caching works correctly.

    **C6: Objective switching**
    - Add two objectives: minimize x, maximize x.
    - Set active to minimize, solve, assert x near lower bound.
    - Switch to maximize, solve, assert x near upper bound.
    - This proves Pitfall 5 (zeroing costs on switch) is handled correctly.
    - Use SetActiveObjective ModelOp for switching.

    **C7: Unsupported rejection** (M1R-H7)
    - Build a ModelSnapshot with a variable that has semicontinuous_lower: Some(2.0).
    - Call synchronize(Synchronization::Rebuild(snapshot)).
    - Assert the result is Err with ErrorCategory::Unsupported.
    - Assert the session health is not modified (the error returns before any HiGHS call).
    - This proves atomic rejection per M1R-H7.

    For each test, use `#[test]` attribute. Since these are integration tests, they run in the roml-highs package.

    NOTE: These tests will run against HighsSession. For tests that also should run against ReferenceBackend, the verification should note that the same test logic can be applied to ReferenceBackend. The primary verification here is against HighsSession.
