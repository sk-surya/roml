<objective>
Establish the foundation for the HiGHS backend rewrite: replace handwritten FFI with authoritative highs-sys bindings, implement fallible lifecycle with typed errors, and define the module structure with thread-safety discipline.

Purpose: The old roml-highs has a 252-line handwritten FFI (ffi.rs), an 886-line monolithic adapter (adapter.rs), and panics on construction failure. After this plan, the crate compiles with its new module structure and fallible lifecycle, ready for projection and session implementation.

Output: A compilable roml-highs crate with bindings, error, lifecycle, and lib.rs modules. Old ffi.rs, adapter.rs, and integration.rs are deleted.
</objective>

  <action>
    Create error.rs:
    - Define constructor helpers for BackendError tailored to HiGHS error codes.
    - A `fn from_native_status(highs_ptr: *mut c_void, op: &str) -> BackendError` helper that queries `Highs_getHighsRunStatus` (or the last error code via `Highs_getModelStatus`) and maps to the appropriate ErrorCategory and HealthEffect.
    - Construction helpers that call `BackendError::new()` and `BackendError::with_code()` from roml::solver::backend. Do not define a new error type — use the frozen BackendError type directly. per AD-1 the public error alias is `pub type HighsError = BackendError;`.
    - A `fn check_highs_status(ret: HighsInt, highs_ptr: *mut c_void, op: &str) -> Result<(), BackendError>` helper: if ret == 0 (STATUS_OK) return Ok(()), otherwise construct a BackendError with the native code set, category mapped from context (default: Internal for unknown error codes), and HealthEffect::Recoverable (caller elevates if needed).

    Create lifecycle.rs:
    - Define the `HighsSession` struct:
      ```rust
      pub struct HighsSession {
          raw: *mut c_void,          // Highs_create handle
          cursor: AdapterCursor,     // from roml::sync
          col_map: IndexMap<VarId>,  // from index_map.rs (kept as-is)
          row_map: IndexMap<ConId>,
          inf: f64,                  // Highs_getInfinity cached value
          // bound/objective caches (extracted from old adapter patterns)
          var_bounds: HashMap<VarId, (f64, f64)>,
          con_bounds: HashMap<ConId, (f64, f64)>,
          obj_costs: HashMap<ObjId, HashMap<VarId, f64>>,
          obj_senses: HashMap<ObjId, Sense>,
          active_obj: Option<ObjId>,
      }
      ```
      Import types from: roml::id::{VarId, ConId, ObjId}, roml::model::objective::Sense, roml::sync::AdapterCursor, crate::index_map::IndexMap.

    - Implement `HighsSession::try_new() -> Result<Self, BackendError>`:
      1. Call `unsafe { highs_sys::Highs_create() }`. Check null pointer.
      2. Call `unsafe { highs_sys::Highs_getSizeofHighsInt(self.raw) }`. If != 4, destroy handle and return Err(BackendError::unsupported("64-bit HighsInt not supported")).
      3. Call `unsafe { highs_sys::Highs_getInfinity(self.raw) }` to cache infinity value as `inf: f64`.
      4. If bundled build, verify version via `Highs_getHighsVersionString` (info-log only, not a fatal error).
      5. Return Ok(Self) with fresh AdapterCursor::new() and empty maps.
      Per M1R-H2, there is NO `fn new()` that panics. If a convenience constructor is needed, name it `new_unchecked()` with doc comment "Panics if HiGHS initialization fails".

    - Implement `Drop for HighsSession`:
      Call `unsafe { highs_sys::Highs_destroy(self.raw) }` only if `self.raw` is not null. Use a flag or check `raw.is_null()`. The safety comment must state: (1) the handle was created by Highs_create in try_new, (2) no other code holds a reference to it, (3) after destruction no further C API calls are made.

    - Implement `Send for HighsSession` with safety comment:
      ```
      // SAFETY: HighsSession owns the *mut c_void handle exclusively.
      // 1. The handle is created in try_new() and destroyed in Drop — no other
      //    code creates or frees it.
      // 2. No internal references to the handle escape the session (all C API
      //    calls are internal, passing raw back to HiGHS functions).
      // 3. Callbacks are set up and torn down within solve() — they do not
      //    survive session moves.
      // Moving the session to another thread is safe because no other thread
      // holds a reference to the handle. However, calling Highs_run concurrently
      // on the same handle from multiple threads is UB — Sync is NOT implemented.
      ```

    - Add a `fn raw_ptr(&self) -> *mut c_void` accessor for internal use by projection/solution modules. Mark it `pub(crate)`.

    - Use `log::info!` / `log::warn!` for lifecycle events (construction success, version info, index width check).
  </action>

|----------|-------------|
| Rust code -> HiGHS C API | Unsafe FFI boundary: every call through highs-sys is an unsafe operation |
| HighsSession across threads | Send boundary: exclusive handle ownership permits Send, but concurrent access is UB |

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-11-01 | DoS | lifecycle.rs: Highs_create null return | high | mitigate | Check raw pointer for null immediately after creation; return Err, never dereference null |
| T-11-02 | Tampering | lifecycle.rs: Highs_destroy double-free | high | mitigate | Drop checks raw.is_null() before calling Highs_destroy; use a bool flag to prevent double-drop |
| T-11-03 | DoS | lifecycle.rs: Highs_getSizeofHighsInt | medium | mitigate | If sz != 4, destroy handle and return Err; do not proceed with wrong index width |
